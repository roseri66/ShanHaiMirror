## 1. MSVC 14.44 不兼容 UE 5.5

### 现象

```
c1xx: fatal error C1083: Cannot open compiler intermediate file: 
"SharedPCH.UnrealEd.Project.ValApi.Cpp20.h.pch": No such file or directory
```

### 原因

UE 5.5 的预编译头（PCH）机制依赖 MSVC **14.38.33130**。VS 2022 默认安装的最新工具链是 14.44，UnrealBuildTool 会自动选择最新可用版本，导致 PCH 编译失败。

### 解决方案

1. 打开 **Visual Studio Installer** → 修改 → 单个组件
2. 搜索并安装：`MSVC v143 - VS 2022 C++ x64/x86 build tools (v14.38-17.8)`
3. 创建 `%APPDATA%\Unreal Engine\UnrealBuildTool\BuildConfiguration.xml`：

```xml
<?xml version="1.0" encoding="utf-8" ?>
<Configuration xmlns="https://www.unrealengine.com/BuildConfiguration">
    <WindowsPlatform>
        <CompilerVersion>14.38.33130</CompilerVersion>
    </WindowsPlatform>
</Configuration>
```

### 验证

```powershell
# 查看已安装的 MSVC 版本
ls "C:\Program Files\Microsoft Visual Studio\2022\Community\VC\Tools\MSVC"
```

确认 `14.38.33130` 目录存在。

## 2. 中文路径导致 MSVC 无法写入中间文件

### 现象

同上 C1083 错误，即使 MSVC 版本正确也无法解决。

### 原因

MSVC 编译器对含中文字符的路径处理有问题，无法在中文路径下创建 `.pch` 预编译头文件和 `.obj` 中间文件。

### 解决方案

将整个 UE5 工程放在**纯 ASCII 路径**下。

❌ 不可用：
```
C:\Users\徐傲轩\Desktop\游戏开发\ShanHaiMirror\
```

✅ 可用：
```
C:\Dev\ShanHaiMirror\
```

### 规则

UE5 工程的**完整路径**中不得包含任何非 ASCII 字符（中文、日文、特殊符号等）。

## 3. 关卡从未保存，所有配置都在临时关卡里，重开就清空

### 现象

角色无法移动/攻击，反复检查了蓝图连线、Input 资产、Class Defaults 都没问题，Standalone 测试还是没反应；用 MCP/Python 连进编辑器查询才发现，当前打开的关卡是 `/Temp/Untitled_1.Untitled_1`——一个从未保存过的临时关卡（还是引擎自带的 Open World 大地图模板），里面没有 GameMode Override、没有自己摆的 PlayerStart/角色/敌人，只有模板自带的东西。

### 原因

之前在编辑器里做的所有 Actor 摆放（PlayerStart、World Settings → GameMode Override、测试用的角色/敌人）都只存在于内存里，没有执行过"保存关卡"。每次重开编辑器、或者不小心新建/切换了关卡，这些配置就全部作废，等于从头再来一遍——但表面上看不出来，容易一直在"检查蓝图/Input 有没有错"这个错误方向上打转。

### 解决方案

1. 文件(File) → 新建关卡(New Level) → 选 **Empty Level**（不要选带地形的模板，比如 Open World）
2. 在这个关卡里摆好 PlayerStart、设置 World Settings → GameMode Override、放测试用敌人等
3. **文件(File) → 保存当前关卡(Save Current Level)**，存到 `Content/Maps/` 下，比如 `L_TestArena` —— 这一步最容易漏掉
4. 编辑(Edit) → 项目设置(Project Settings) → Maps & Modes，把 **Editor Startup Map** 和 **Game Default Map** 都设成这个关卡，这样以后开编辑器/点 Play 都不会再跳回临时关卡

### 验证

用 Python 查 `unreal.EditorLevelLibrary.get_editor_world().get_path_name()`，确认路径是 `/Game/Maps/L_TestArena` 这种正式资产路径，不是 `/Temp/Untitled_*`。

## 4. Enhanced Input：Character 的 BeginPlay 里 AddMappingContext 不生效

### 现象

IMC/IA 资产都在 Class Defaults 里正确赋值了，BeginPlay 蓝图连线也没问题（Cast To PlayerController → Add Mapping Context），但 WASD、攻击键完全没反应；加了 Print String 测试证明 BeginPlay 确实执行了，问题更隐蔽。

### 原因

对于 GameMode 通过 PlayerStart 生成的 Pawn，引擎的实际调用顺序是：**Pawn::BeginPlay 先执行，GameMode 之后才调用 Controller->Possess(Pawn)**。也就是说 Character 的 BeginPlay 触发时，`Get Controller` 返回的还是 None，`Cast To PlayerController` 必然失败，`Add Mapping Context` 从未真正跑起来。

（如果之前用"手动在关卡里摆一个 Pawn，勾 Auto Possess Player = Player 0"的方式测试过，那种情况下 Possess 是发生在 BeginPlay 之前的，顺序是反的，看起来蓝图连线"是好的"，一换成正规 GameMode 流程就失效，很容易把排查方向带偏。）

### 解决方案

把加载 Mapping Context 的逻辑从 **Event BeginPlay** 换成 **Event Possessed**（Pawn 自带的蓝图事件，专门在被 Controller 占有的那一刻触发，自带 `New Controller` 输出引脚）：

```
Event Possessed (New Controller)
  → Cast To PlayerController (Object = New Controller)
    → Get Enhanced Input Local Player Subsystem
      → Add Mapping Context (IMC_Default, Priority = 0)
```

### 规则

任何依赖"Pawn 被哪个 Controller 占有"的初始化逻辑（尤其是 Enhanced Input 绑定），都不要放在 Pawn 的 BeginPlay 里，放 `Event Possessed` 或者 Controller 自己的 BeginPlay 更安全。

## 5. 属性结构体 Base 默认值为 0，角色/敌人出生即 0 血

### 现象

关卡/GameMode/Input 全部配置正确、角色也能操作了，但血量表现异常。

### 原因

`FSHMAttribute::Base` 在 C++ 里默认是 `0.f`，而 `USHMAttributeComponent::BeginPlay()` 里执行 `CurrentHP = GetMaxHP()`。如果没有人显式给 `Health.Base` 赋值（比如没有走 `InitFromDataTable` 这条初始化路径），运行时血量就是 0。

### 解决方案

在角色/敌人的 Blueprint Class Defaults 里，展开 `AttributeComp` 组件，手动把 `Health.Base` 设成一个非零值（例如玩家 100，杂兵 80）。

### 规则

凡是 Base 默认 0 的属性结构体，新建一个用到它的 Character/Enemy Blueprint 时，第一件事就是去 Class Defaults 里检查有没有忘记填初始值，不要假设"组件存在就等于数值配好了"。

## 6. `ASHMEnemy::OnOwnerDeath()` 空指针崩溃（BrainComponent 为空）

### 现象

玩家攻击打死一个敌人的瞬间，编辑器直接崩溃闪退，没有报错弹窗，只留下崩溃日志（`Saved/Crashes/`）。

### 原因

`OnOwnerDeath()` 里写了 `AIC->BrainComponent->StopLogic("Dead")`，但 `BrainComponent` 只有在 AIController 真正跑过 `RunBehaviorTree()` 之后才会被引擎赋值。Sprint 1 明确不用行为树（AI 直接在 Tick 里 `MoveToActor` 追玩家），`BrainComponent` 永远是 `nullptr`，一调用 `StopLogic` 就野指针崩溃。

崩溃堆栈关键帧：

```
ASHMEnemy::OnOwnerDeath() [SHMEnemy.cpp:65]
← USHMAttributeComponent::ApplyDamage()
← ASMCharacter::PerformMeleeAttack()
← ASMCharacter::OnAttack()
```

### 解决方案

调用前判空：

```cpp
if (AIC->BrainComponent)
{
    AIC->BrainComponent->StopLogic("Dead");
}
```

### 规则

任何"预留给未来功能"的代码（这里是预留给以后可能接入的行为树），只要当前功能没启用，相关指针大概率是空的，使用前一定要判空，不能假设它一定被赋值过。

## 7. 蓝图 Class Defaults 改了，关卡里已经摆好的 Actor 实例却没变

### 现象

把 `BP_Grunt` 的 AI Controller Class 在蓝图 Class Defaults 里改成新的类、编译保存后，Standalone 测试敌人还是没反应；查询关卡里已摆放的敌人实例，AI Controller Class 显示的还是老值。

### 原因

Actor 一旦被摆放/生成到关卡里，它自己的属性会独立存一份（instance data），不会随着后续对蓝图 Class Default（CDO）的修改自动同步。改 CDO 只影响"以后新生成"的实例，已经在关卡里的旧实例必须单独处理。

### 解决方案

两种做法任选：

1. 把关卡里旧的实例删掉，重新从改好的蓝图拖新的实例进去
2. 或者直接选中场景里的每个旧实例，在 Details 面板里把对应属性重新赋值一遍，保存关卡

### 规则

改了蓝图的 Class Defaults 之后，如果关卡里已经有这个蓝图的实例，一定要额外确认/重新摆放这些实例，不能默认它们会自动同步。

## 8. UnrealBuildTool 编译时因 Content 目录里的中文文件夹名而崩溃退出

### 现象

用 `Build.bat` / UnrealBuildTool 编译 Editor Target 时，还没开始真正编译就直接抛未处理异常退出：

```
Unhandled exception. System.ArgumentException: Path fragment '"UnrealProject/Content/\346\214\211\351\224\256\346\223\215\344\275\234\347\273\221\345\256\232/"' contains invalid directory separators.
   at ... UnrealBuildTool.GitSourceFileWorkingSet.AddPath(...)
```

### 原因

UBT 有个"Adaptive Non-Unity Build"相关功能，会跑一次 `git status --porcelain` 来判断哪些文件是"最近改动过的工作集"。项目 Content 目录下如果有中文命名的文件夹（例如"按键操作绑定"），git 默认的 `core.quotepath=true` 行为会把这种非 ASCII 文件名在 status 输出里转成带反斜杠转义的八进制形式，UBT 自带的解析器不认这种转义格式，直接抛异常导致整个编译进程崩掉。

注意这个坑和"1. MSVC 不兼容""2. 中文路径"是不同机制：这里工程本身路径是纯 ASCII 的（`C:\Dev\ShanHaiMirror`），问题出在 **Content 子目录里出现了中文命名**。

### 解决方案

编辑 `%APPDATA%\Unreal Engine\UnrealBuildTool\BuildConfiguration.xml`（这是 UBT 的个人偏好文件，不是 git 配置，不影响仓库和团队其他人），加一段关掉 Git working-set 检测：

```xml
<SourceFileWorkingSet>
    <Provider>None</Provider>
</SourceFileWorkingSet>
```

### 规则

项目 Content 目录（或任何会被 git 追踪到的路径）里也尽量别用中文文件夹/文件名命名，能从根源避免这个坑，也顺便避免以后接入其他依赖 `git status` 解析的工具（比如 CI、其他构建加速工具）时踩同样的雷。

## 9. UFUNCTION 参数名叫 `Instigator`，UHT 直接报错

### 现象

给 `ASHMEnemy` 加了一个受击回调，编译在 UnrealHeaderTool 阶段就挂了，连 cl.exe 都没跑起来：

```
SHMEnemy.h(62): Error: Function parameter: 'Instigator' cannot be defined in 'OnDamaged'
as it is already defined in scope 'AActor' (shadowing is not allowed)
```

### 原因

`AActor` 自带一个 `Instigator` 成员（`APawn* Instigator`）。UHT 禁止 `UFUNCTION` 的参数名遮蔽所属类作用域里已有的成员，且项目编译带 `-WarningsAsErrors`，警告即错误。

**容易被带偏的地方**：同一份代码里 `USHMAttributeComponent::ApplyDamage(float Amount, AActor* Instigator, ...)` 用同样的参数名却完全没事——因为 `UActorComponent` 没有 `Instigator` 成员。所以"这个名字明明别处能用"的直觉在这里是错的，**决定因素是所属类的继承链**。

### 解决方案

换个参数名，比如 `DamageInstigator`：

```cpp
UFUNCTION()
void OnDamaged(AActor* DamageInstigator, float Damage);
```

注意只有 `UFUNCTION`（要被 UHT 解析的）受这条约束，普通 C++ 成员函数和 `DECLARE_DYNAMIC_MULTICAST_DELEGATE` 的参数名不受影响——所以委托声明里叫 `Instigator` 是可以的，绑定它的那个 `UFUNCTION` 却不行。

### 规则

在 `AActor` 子类里写 `UFUNCTION` 时，避开 AActor 已有的常见成员名：`Instigator`、`Owner`、`Role`、`InputComponent`、`Children`。拿不准就加个前缀。

## 10. UE 5.5 把 `EAutomationTestFlags` 改成了 enum class，网上示例全部编译不过

### 现象

照着教程写第一个自动化测试，编译报：

```
error C2440: “return”: 无法从“const int32”转换为“EAutomationTestFlags”
```

出错位置指向 `IMPLEMENT_SIMPLE_AUTOMATION_TEST` 宏展开处，不是自己写的那一行，很难一眼看出是 flags 的类型问题。

### 原因

UE 5.4 及以前，`EAutomationTestFlags` 是一个 **namespace**，里面是 `uint32` 常量，所以社区里通行的写法是：

```cpp
const int32 TestFlags = EAutomationTestFlags::EditorContext | EAutomationTestFlags::EngineFilter;  // 旧写法
```

UE 5.5 起它变成了 **enum class**（`Runtime/Core/Public/Misc/AutomationTest.h:83`，并有 `ENUM_CLASS_FLAGS`），`int32` 不再能隐式转回枚举类型。绝大多数教程和 AI 生成的示例代码还停留在旧写法。

### 解决方案

直接用枚举类型声明：

```cpp
constexpr EAutomationTestFlags TestFlags =
    EAutomationTestFlags::EditorContext |
    EAutomationTestFlags::EngineFilter;
```

顺带一提，同版本还提供了 `EAutomationTestFlags_ApplicationContextMask` 这类预组合常量（注意是**下划线**前缀的全局 constexpr，不是枚举成员），需要跨 context 时用它。

### 规则

UE 大版本升级后，凡是编译报"无法从 int/uint32 转换为 EXxx"的，先去引擎头文件里确认那个 `EXxx` 是不是从 namespace 常量改成了 enum class，不要在自己的调用代码上反复试。

## 11. 命令行跑自动化测试：没有 `-stdout` 就一行输出都拿不到；测试名用中文会变乱码

### 现象

跑命令行自动化测试，进程返回 255，但终端一个字都没输出，无从判断是测试失败还是根本没跑起来：

```bash
UnrealEditor-Cmd.exe <uproject> -ExecCmds="Automation RunTests SHM.Director;quit" -unattended -nopause
# exit=255，stdout 空
```

### 原因

两个独立的问题叠在一起：

1. `UnrealEditor-Cmd.exe` 默认把日志写进 `Saved/Logs/*.log`，**不往 stdout 打**。要拿到终端输出必须显式加 `-stdout`。
2. 退出码 255 是**测试失败的正常表现**（引擎会 `Setting GIsCriticalError due to test failures`），不代表启动失败。全部通过时退出码是 0。第一次跑很容易把"有测试没过"误判成"命令写错了"。

### 解决方案

```bash
UnrealEditor-Cmd.exe "<uproject>" \
  -ExecCmds="Automation RunTests SHM.Director;quit" \
  -unattended -nopause -nosplash -nullrhi \
  -stdout -AbsLog="<项目>\Saved\Logs\AutoTest.log"
```

- `-nullrhi` 免掉起渲染设备，跑纯逻辑测试快很多
- `-AbsLog` 固定日志路径，方便脚本 grep，不用每次去找带时间戳的文件名
- 判成功看这一行：`**** TEST COMPLETE. EXIT CODE: 0 ****`

### 附带的坑：测试名不要用中文

`IMPLEMENT_SIMPLE_AUTOMATION_TEST` 的**测试路径**如果写中文，在自动化日志和报告里会变成乱码：

```
Path={SHM.Director.ProfileAnalyzer.￥ﾍﾕ￤ﾸﾀ￦ﾭﾦ￥ﾙﾨ...}
```

但 `TestEqual(TEXT("中文描述"), ...)` 里的**断言消息**显示是正常的。所以正确做法是：

- 测试路径用 ASCII（`SingleWeapon_FullConcentration`）
- 断言消息用中文（`只用一把武器时集中度应为 100`）

两边都好读，日志也不会花。

### 规则

自动化测试的标识符（测试路径、类名）一律 ASCII，人读的描述文字才用中文。这条同样适用于以后接 CI——工具链对非 ASCII 标识符的处理普遍不可靠。

## 12. Standalone 下玩家开局不能动，必须先点一下鼠标（视口没拿到键盘焦点）

### 现象

跑 Standalone，开局玩家用 WASD 走不动，像卡住了；用鼠标**左键（攻击）或右键（闪避）点一下之后**，键盘立刻恢复正常，之后一直正常。同时敌人靠近后站着不动（这部分是另一回事，见末尾）。

极易误判的干扰点：

- **鼠标中键（没绑任何功能）点窗口不解冻** → 排除了"点窗口给焦点"这种笼统解释
- **任何键盘键都不解冻** → WASD 本身就是被冻的对象
- **重启编辑器无效** → 排除了 CPU 降频（`bThrottleCPUWhenNotForeground`）那类猜测
- 之前在 PIE（编辑器视口内 Play）测试一直正常，只有 Standalone 复现

### 原因

`ASMPlayerController` 设了 `bShowMouseCursor = true`，但**没有配套调用 `SetInputMode`**。光标显示且未设输入模式时，游戏视口（SViewport）启动时可能不持有键盘焦点，键盘事件（`IA_Move`）到不了 Pawn。

为什么左右键能解冻、中键和键盘不能：`bEnableClickEvents = true` 时，左/右键会走视口的点击事件处理路径，**顺带把键盘焦点交给视口**；中键不参与点击事件处理、键盘键在没焦点时直接被丢弃，所以都不触发这次焦点移交。焦点一旦交给视口就留在那儿，故"点一次之后一直正常"。

PIE 不复现是因为编辑器视口本来就有焦点、会把焦点传给游戏；Standalone 是独立窗口，才暴露这个缺失。

### 解决方案

在 `PlayerController::BeginPlay` 里显式设置输入模式，强制视口开局持有输入焦点：

```cpp
void ASMPlayerController::BeginPlay()
{
    Super::BeginPlay();

    FInputModeGameOnly InputMode;
    SetInputMode(InputMode);   // 全部输入路由给游戏；bShowMouseCursor=true 仍保留光标
}
```

`FInputModeGameOnly` 与 `bShowMouseCursor = true` 并不冲突：光标照常显示，`GetHitResultUnderCursor`（瞄准/攻击方向）照常工作，这正是俯视角 twin-stick 想要的组合。

### 规则

任何设了 `bShowMouseCursor = true` 的 PlayerController，必须在 `BeginPlay` 里配一个 `SetInputMode`（GameOnly 或 GameAndUI），否则视口开局可能没有键盘焦点。**只在 PIE 里测不出来这个坑，交付前一定要用 Standalone 或打包版本验一遍输入。**

### 附：这次排查踩的方法论坑（值得单独记）

一开始因为"敌人也像是冻住"，误判成整个引擎降频，加了 `bThrottleCPUWhenNotForeground=False` 又删掉——白走一趟。真相是**敌人根本没冻**：它们到达 `AcceptanceRadius` 停下、且攻击行为未实现，所以站着不动，跟玩家输入毫无关系。教训：多个症状不一定同源，"敌人不动"和"玩家不动"是两件事，把它们绑成一个根因导致方向跑偏。真正劈开问题的是那个反直觉的观测——**中键（无绑定）点窗口不解冻**，它一举排除了"焦点=点窗口"和"降频"两条错误路径。遇到卡壳，去找能一刀切开假设的那个观测，而不是继续在某一条假设上加码。
